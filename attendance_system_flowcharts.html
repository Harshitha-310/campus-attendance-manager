Flowchart 2 — Attendance Marking Flow (with Triggers)
Core transaction of the system. Two triggers fire automatically — one before insert (validation) and one after insert (monitoring).
Professor selects section, slot, date
UI auto-shows: course name, department, coordinator
Load Students button clicked
GET /api/sections/{id}/students → query joins student + registration
Professor marks P or A for each student
Default is P. Click to toggle. All Present / All Absent shortcuts.
Submit Attendance clicked
POST /api/attendance/mark-bulk → calls attendance_pkg.mark_attendance()
Oracle Trigger 1: prevent_dup_attendance — BEFORE INSERT
Check 1: Is student registered in this section?
SELECT COUNT(*) FROM registration WHERE student_id = :NEW.student_id AND section_id = :NEW.section_id
NO — not registered
RAISE_APPLICATION_ERROR(-20001)
INSERT blocked. Error sent to frontend.
YES — registered
Check 2: Already marked today?
Same student + section + slot + date already in table?
YES — duplicate
RAISE_APPLICATION_ERROR(-20002)
INSERT blocked.
NO — new record
INSERT succeeds
Attendance row written to DB
Oracle Trigger 2: check_attendance_eligibility — AFTER INSERT
Calculate attendance % for student in section
COUNT total rows, SUM where status='P', compute ROUND(present/total×100, 2)
pct < 75%
Log WARNING
Insert row into ATTENDANCE_LOG with action_taken = 'WARNING: ...'
pct ≥ 75%
Log OK
Insert row into ATTENDANCE_LOG with action_taken = 'OK: ...'
React shows success toast
All student records processed. Errors shown if any row failed.
Flowchart 3 — Report Generation Flow
Three types of reports — by section, by student, or all. Each path queries the same underlying joined tables.
User opens Reports page
Admin or Professor role — students see My Attendance instead
Select report type
All Students · By Section · By Student
All Students
GET /reports/all
Join: student + dept + registration + course + section + professor + attendance
By Section
GET /reports/section/{id}
Same joins filtered by section_id
By Student
GET /reports/student/{id}
Same joins filtered by student_id
Oracle calculates per row
COUNT(att_date) = total · SUM(P) = present · ROUND(present/total×100,2) = pct
CASE statement determines eligibility
pct ≥ 75 → ELIGIBLE · pct < 75 → NOT ELIGIBLE
React renders table
Student · Dept · Course · Section · Coordinator · Total · Present · % · Status
Click View Dates
Date-wise modal opens
GET /attendance/datewise?student_id=&section_id= Shows each date + slot with P/A pill
Click Export CSV
CSV downloaded
Browser creates Blob from filtered table data. Auto-downloads file.
Dashboard charts
Bar + Pie charts update
Filter by course or section. Green bars = eligible, red = not eligible.
